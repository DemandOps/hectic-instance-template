# Hectic Server Deployment

Canonical production deployment guide.

Architecture: systemd + Caddy + PostgreSQL with blue-green app slots.

## What Gets Installed

| Component | Purpose | Port |
|-----------|---------|------|
| **Caddy** | Reverse proxy + automatic HTTPS | 80, 443 |
| **Hectic Blue** | App slot A | 8080 (localhost) |
| **Hectic Green** | App slot B | 8081 (localhost) |
| **Prometheus** | Metrics collection | 9090 (localhost) |
| **Grafana** | Dashboards | 3000 (localhost) |
## Quick Start (Fresh Server)

```bash
# 1) Create a private customer instance repo from the public Hectic instance template
#    and open that repo in your agent or terminal.

# 2) SSH into Ubuntu 22.04+ host
ssh root@your-server-ip

# 3) Clone the instance repo and run bootstrap
git clone git@github.com:<your-org>/<your-hectic-instance-repo>.git /opt/hectic-instance
cd /opt/hectic-instance/deploy
sudo ./setup.sh

# 4) Fill in hectic.instance.yaml and GitHub secrets in the instance repo.
#    The deploy workflow renders /opt/hectic/.env from those inputs.
#    Manual editing of /opt/hectic/.env should be treated as a break-glass path only.

# 5) Configure deploy SSH key for user 'hectic'
mkdir -p /opt/hectic/.ssh
# append deploy public key to /opt/hectic/.ssh/authorized_keys
chmod 600 /opt/hectic/.ssh/authorized_keys
chown -R hectic:hectic /opt/hectic/.ssh

# 6) Trigger production deploy from the instance repo
```

The deploy workflow will:

- read `hectic.instance.yaml`
- pull the pinned core artifacts
- render `/opt/hectic/.env` from config plus secrets
- run migrations
- deploy to the inactive slot
- health-check it
- switch the active slot

The core source repo is still where Hectic itself is developed. The customer-facing production path should use a private instance repo as the deployment control plane so core upgrades stay pinned to released artifacts instead of long-lived customer forks.

## Runtime Environment Contract

The runtime environment file at `/opt/hectic/.env` is normally generated by the instance repo workflow.

If you need to debug it manually, it should look like:

```bash
# App
ENVIRONMENT=production
DOMAIN=hectic.example.com
DATABASE_DSN=postgres://hectic:<password>@postgres.example.com:5432/hectic?sslmode=require

# Auth
JWT_SECRET=<openssl rand -base64 64>
JWT_EXPIRY=24h
MAGIC_LINK_TTL=15m
COOKIE_DOMAIN=.hectic.example.com

# Email
EMAIL_BACKEND=postmark
FROM_EMAIL=hectic-noreply@example.com
FROM_NAME=Example Support
POSTMARK_SERVER_TOKEN=<token>
POSTMARK_WEBHOOK_SECRET=<openssl rand -hex 32>

# Or SMTP
# EMAIL_BACKEND=smtp
# SMTP_HOST=smtp.example.com
# SMTP_PORT=587
# SMTP_USERNAME=<username>
# SMTP_PASSWORD=<password>

# Or SES
# EMAIL_BACKEND=ses
# SES_REGION=us-east-1
# SES_ACCESS_KEY=<key>
# SES_SECRET_KEY=<secret>

# Storage / S3
STORAGE_TYPE=s3
STORAGE_PATH=/opt/hectic/data
S3_REGION=ams3
S3_ENDPOINT=https://ams3.digitaloceanspaces.com
S3_ATTACHMENTS_BUCKET=hectic-attachments
AWS_ACCESS_KEY_ID=<key>
AWS_SECRET_ACCESS_KEY=<secret>

# Runtime
CACHE_ENABLED=true
CACHE_TTL=1h
RATE_LIMIT_ENABLED=true
RATE_LIMIT_PER_MIN=60
METRICS_TOKEN=<openssl rand -hex 32>
TRUSTED_PROXIES=127.0.0.1,::1
```

## DNS

Point these domains to your server IP:

- `hectic.example.com`
- `admin.hectic.example.com`
- `api.hectic.example.com`

Caddy obtains certificates automatically.

## Service Management

```bash
# Active slot
cat /opt/hectic/.active-slot

# Status
systemctl status hectic-blue hectic-green caddy prometheus grafana-server

# Logs
journalctl -u hectic-blue -f
journalctl -u hectic-green -f
journalctl -u caddy -f

# Restart current slot manually (example)
systemctl restart hectic-blue
```

## Verification

```bash
# Internal
curl http://127.0.0.1:8080/health
curl http://127.0.0.1:8081/health
curl http://127.0.0.1:9090/-/healthy
curl http://127.0.0.1:3000/api/health

# External
curl https://api.hectic.example.com/health
curl -I https://admin.hectic.example.com/login
```

## Backup & Restore

PostgreSQL backup, point-in-time recovery, and restore are owned by your
database provider or your PostgreSQL operations runbook. The Hectic deploy
repo verifies connectivity through `DATABASE_DSN` only.

## Troubleshooting

### Service won't start

```bash
journalctl -u hectic-blue -n 100 --no-pager
journalctl -u hectic-green -n 100 --no-pager
```

### Grafana not loading

1. Check `curl http://127.0.0.1:3000/api/health`
2. Verify Caddy `/grafana/*` route
3. Confirm Grafana embedding settings in `/etc/grafana/grafana.ini`

### Database connectivity

```bash
set -a
source /opt/hectic/.env
set +a
psql "$DATABASE_DSN" -c "SELECT current_database(), current_user;"
```

## Security Notes

- Prometheus and Grafana bind to `127.0.0.1` only.
- All external traffic is TLS-terminated in Caddy.
- Keep `/opt/hectic/.env` mode `600` and owned by `hectic`.
